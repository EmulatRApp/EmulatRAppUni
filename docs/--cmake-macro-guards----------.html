CMake Macro Guards

A number of EmulatR behaviors are selected at build time through CMake options that define compile-guard macros, rather than at run time. This keeps diagnostic and debug scaffolding out of the shipping hot path: a guard that is off compiles to nothing. The options are set in the CMake configuration (for example through Qt Creator) and define the matching EMULATR_* macro for the build.

 

Build / behavior options

Guard

Effect

EMULATR_DEBUG_STEP

Restores an -O0 build of the per-instruction pipeline step for single-step debugging. Off by default so the hot loop ships optimized.

EMULATR_FP_SOFTFLOAT

Selects the SoftFloat compute backend for floating point instead of the native host path, for bit-exact host-independent results.

EMULATR_USE_OS_PAGES

Selects the OS-page (VirtualAlloc / mmap) backing mode for guest memory.

 

Diagnostic / trace guards

Guard

Effect

EMULATR_TRACE_HOOKS

Compiles in the per-step trace hooks used to capture an execution trace.

EMULATR_MEMDIAG

Compiles in the memory-diagnostic instrumentation (the cycle-windowed fetch / load / store dumps).

EMULATR_CHIPSET_DIAG

Enables chipset-level diagnostic output.

EMULATR_DIAGNOSTIC_LOGGING

Enables the broader diagnostic logging path.

EMULATR_PA_DUMP

Compiles in physical-address dump instrumentation.

EMULATR_BOOTSTRAP_ITB_BYPASS

Debug only: bypasses the ITB for the known reset / PAL entry page during bring-up. Must be removed before relying on permission bits or ASN isolation.

EMULATR_TRACE_CONSOLE / EMULATR_ISADIAG

Console-trace and ISA-diagnostic channels.

 

The default build leaves the diagnostic guards off; they are turned on only when chasing a specific problem, because the instrumentation they add (especially the memory and trace hooks) is heavy enough to change runtime substantially. Runtime behavior that does not need a recompile is controlled instead by the EMULATR_* environment variables.
