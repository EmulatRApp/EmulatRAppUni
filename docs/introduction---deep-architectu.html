Overview

EmulatR is a full-system, high-fidelity emulator of the DEC Alpha AXP platform -- specifically the 21264 (EV6) processor and the 21272 (Tsunami / Typhoon) core logic as integrated in AlphaServer DS10 / ES40 / ES45-class systems. It is implemented in modern C++ (C++20) with a deliberately framework-free architectural core (Qt is confined to the periphery -- console UI and host glue), built with CMake. Its purpose is not to approximate an Alpha; it is to be one, faithfully enough that unmodified DEC SRM console firmware, PALcode, and the operating systems that ran on the metal run on EmulatR without knowing the difference.

 

This page describes the architecture of the current implementation (V4) -- the execution model, the memory and translation system, the privileged boundary, and the device fabric -- and the standard every subsystem is measured against. The chapters that follow expand each area; the deep-dive appendices carry the register-level detail.

 

The fidelity standard and its authoritative sources

EmulatR is governed by one standard: it must behave like real Alpha hardware in every architecturally visible way, even when that complicates the implementation. Correctness is defined by primary sources -- the Alpha Architecture Reference Manual and the original DEC SRM console and PALcode sources. These define what correct behavior is.

 

The test is black-box and adversarial, conducted by the most demanding guests available: the SRM console and PALcode exercise the privileged architecture directly, and a booting operating system exercises the memory model, the trap machinery, and the device fabric under real workloads. A second emulator (AXPBox) is consulted only as a convenience cross-check for differential triage -- it is explicitly not an authoritative source, because another implementation can be wrong, whereas the architecture manual and the firmware sources cannot. If a guest can distinguish EmulatR from an AlphaServer through architecturally defined behavior, the emulator is wrong.

 

Primary goals and objectives

1. Architectural correctness first. EmulatR models Alpha's architectural semantics, not a particular chip's timing. Correctness always outranks speed.

2. Deterministic, debuggable execution. Execution is reproducible and inspectable. The run loop is staged and cycle-counted, and the hot path is deliberately shallow so a stack trace at any point stays short and legible.

3. Explicit, per-instruction semantics. Every instruction resolves to one faithful grain leaf; behavior is never inferred from an operand-shape family. Architectural results and exceptions become visible only at the Writeback stage.

4. Separation of concerns. Decode, integer, floating-point, memory, control, and privilege are isolated into well-defined functional domains -- the "Boxes" -- with one-way dependency flow. No domain reaches into another's internal state.

5. SMP-aware by construction. Per-CPU state, CPU-local translation buffers, and load-locked / store-conditional reservation tracking are built in, so a multiprocessor build is an extension rather than a rewrite (see the scope note below).

 

The execution model

Execution is driven by PipelineDriver (pipelineLib) through six in-order stages: IF (instruction fetch), DE (decode / dispatch), GR (grain-ready: read the register file into the execution context), EX (execute), MEM (memory drain), and WB (writeback). Each instruction is carried through the six stages on a PipelineSlot; architectural state -- the register file and the PC -- is committed only at WB, the single well-defined commit point that makes exceptions precise and replay deterministic.

 

Dispatch is flat. At DE a decoded instruction is resolved through the grain system to a single grain leaf -- a pure function with the signature BoxResult(InstructionGrain const&, ExecCtx const&) -- that implements its semantics. Execution is partitioned into the functional Boxes, each a namespace of grain leaves rather than a class hierarchy: IBox (control flow -- branches, jumps), EBox (integer ALU), FBox (floating point, IEEE and VAX, over the fpBox backend), MBox (loads, stores, load-locked / store-conditional), CBox (branch prediction and target resolution), and PalBox (the privileged boundary). Dependencies flow one way: every Box depends on coreLib; most also on iprLib; MBox additionally on memoryLib.

 

A deliberate consequence is that the hot path is shallow -- from the driver, through the stage, into a grain leaf, the call-stack depth is typically two and rarely exceeds four. That keeps traces legible (serving debuggability) and minimizes the per-instruction scaffolding that profiling has shown -- rather than the opcode bodies -- to dominate runtime (serving performance without compromising correctness). The grain leaves are the leaves of that call tree by construction.

 

Dispatch tables are generated, not hand-maintained. GrainMasterV4.tsv is the single source of truth for instruction semantics; the genGrains.py codegen reads it and emits the dense primary table (indexed by the 6-bit opcode) plus the sparse secondary tables for multi-function opcodes, including the personality-aware PALcode dispatch (OSF/1 and VMS). There is no runtime trace cache or JIT; the generated tables are the decode artifact.

 

Memory and address translation

Physical backing. GuestMemory (memoryLib) is the single physical-memory interface -- a passive byte-store that pages the guest's physical address space in on demand, in 64 KiB pages, committed only as the guest first touches them, so a multi-gigabyte machine costs only its working set. The pages are backed by host VirtualAlloc on Windows and by mmap on Linux / ARM hosts (with a portable fallback). The earlier SafeMemory layer has been removed. GuestMemory itself does no MMIO routing or fault generation: the TsunamiChipset system-bus arbiter decodes addresses, routes device traffic, and raises range faults before raw RAM is ever touched. A LockMonitor tracks per-CPU load-locked / store-conditional reservations at cache-line granularity.

 

Guest translation. Address translation is performed by Ev6Translator (mmuLib) as the EV6 architecture specifies: PAL-mode bypass, the canonical virtual-address window check, and superpage (kseg) mapping, with non-mapped accesses returning an ITB or DTB miss. The translation buffers are modeled by the SPAM cache -- Set-Partitioned Associative Memory -- as SPAMShardManager instances resident in CpuState (itbMgr and dtbMgr, each 16 shards by 8 ways = 128 entries). Invalidation is epoch-based and lazy (TlbEpoch): a TBIA bumps a global counter and a TBIAP a per-process one, so a flush is O(1) and stale entries simply fail the liveness test at lookup. The translation buffers are CPU-local by design.

 

Faults, exceptions, and the privileged boundary

There is no central fault dispatcher. A leaf, or the memory drainer that detects a fault, sets a faultCode on the instruction's BoxResult (for example a DTB miss or an unaligned access). The hardware exception entry points are compile-time constants in Ev6EntryVectors.h (the vectors at 0x100 through 0x780, spaced 0x80 apart). At WB the driver inspects the fault: a halt sets the CPU's halted flag; otherwise, once PAL_BASE is established, control diverts to the computed entry vector. Privileged operations -- CALL_PAL, HW_MTPR / HW_MFPR, HW_REI -- are handled in PalBox, which applies IPR side-effects (PAL_BASE normalization, TLB invalidation, ASN context) inline. Internal processor registers are flat members of CpuState.

 

The machine and its device fabric

Machine (systemLib) is the top-level orchestrator: it owns the CpuState, the GuestMemory, and the TsunamiChipset, and provides firmware load, reset, and the step / run entry points. The chipset aggregates the three 21272 components -- the Cchip (system control, memory arbitration, interval timers, DRAM array configuration), the Dchip (data path), and the Pchip (PCI host bridge) -- together with the on-board devices. The PCI fabric is a standard bus / slot / function hierarchy (there is no separate "stratum" abstraction).

 

The southbridge is the Cypress CY82C693 PCI-to-ISA bridge (Cypress_CY82C693ISABridge), which carries the ISA / SuperIO plane. The modeled devices include the CY82C693 IDE / ATAPI channel, the SuperIO controller (FDC / COM / LPT), the TOY / RTC, a 16550 UART for the serial console, the PCF8584 IIC controller, the flash / SROM, and the dual 8259A interrupt controllers (Pic8259Pair). SCSI endpoints (VirtualScsiDevice and the ISO-CD variant) exist in the device library.

 

Lineage: four implementations, one set of objectives

The current emulator is the fourth implementation; the objectives above were fixed from the start, and each version reached for them by a different mechanism.

•V1 dispatched each instruction through a virtual call to a per-instruction grain. It was correct and booted SRM, but the per-call overhead capped throughput.

•V2 chased performance by dispatching on instruction families (operand-shape signatures). It was fast, but families under-determined behavior -- instructions sharing a signature still needed per-instruction corner-case handling the model could not express.

•V3 tried to hybridize V1 and V2; the coupling was too tight and introduced silent-failure modes -- the worst outcome for an emulator whose value is fidelity.

•V4 keeps a flat dispatch path but resolves it to faithful per-instruction grain leaves via generated tables, with performance pursued as the sum of many marginal gains rather than one architectural bet. The dispatch mechanism evolved across all four; the objectives did not.

 

Scope note: this page describes the V4 architecture and the standard it is held to. Some elements are infrastructure ahead of full activation -- multiprocessor execution and cross-CPU IPI, full PAL trap delivery, and SCSI / PCI device enumeration.
