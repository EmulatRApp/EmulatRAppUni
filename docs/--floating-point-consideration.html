Floating-Point Considerations

Alpha floating point is wider than a single IEEE implementation, and modeling it faithfully has several moving parts: four data formats, a separate enable gate, a control register that selects rounding and trap behavior, and a choice of compute backend. This page records the considerations a developer needs when reading or extending the fBox; it is a design overview rather than a per-opcode status table, because the instruction coverage is actively being filled in.

 

Four formats

Alpha supports two IEEE formats -- S (single) and T (double) -- and the VAX formats F, G, and D. The distinction matters for guests: OpenVMS Alpha uses VAX G_float as its default double and F_float as its default single, while Tru64 and IEEE workloads use S and T. Data movement is format-complete: the loads and stores for all four memory formats (LDF/LDG/LDS/LDT and STF/STG/STS/STT) move values correctly, so a guest can hold VAX G_float values in registers. The work that distinguishes the formats is the arithmetic and the conversions between them, which is where build-out effort concentrates.

 

The FP-enable gate

Floating point is gated by the FEN (FP-enable) bit. The PAL path for it is in place: MTPR_FEN and MFPR_FEN move the bit, CLRFEN clears it, and an attempt to execute FP while disabled delivers through the FEN entry vector at offset 0x200. So the enable / disabled-trap machinery works independently of how much arithmetic is implemented behind it.

 

FPCR, rounding, and traps

The floating-point control register (FPCR) selects the dynamic rounding mode and the IEEE trap-enable bits; instruction qualifiers select trap mode and rounding per-instruction. Faithful FP must honor these rather than always rounding to nearest and never signaling. The implementation approach uses the host's <cfenv> facilities -- fesetround, feclearexcept, fetestexcept -- around a native host operation to realize the FPCR rounding mode and to detect and deliver IEEE exceptions. Move-to / move-from FPCR provide the storage; the qualifier handling is what connects it to each operation.

 

The backend abstraction

FP arithmetic is reached through a backend interface, IFpBackend (in fpBoxLib), so the grain leaves call an abstract operation set -- the IEEE S/T and VAX F/G add / subtract / multiply / divide / square-root, the compares, and the format conversions -- without binding to one implementation. Operands and results are passed as raw register images (uint64_t) plus an FpExecCtx that carries the rounding mode and qualifiers, and an FpResult that carries the value and the exception flags. Wiring this interface into every FP grain is the in-progress integration step.

 

Berkeley SoftFloat -- the deterministic oracle

The reference implementation of IFpBackend is SoftFloatBackend, built on the Berkeley SoftFloat Release 3 library (vendored in the tree as berkeley-softfloat-3-master and compiled into a softfloat static library, 8086-SSE specialization). It is the deterministic, bit-identical-across-hosts FP truth source: the rounding mode is passed to SoftFloat as data and the exception flags are harvested as data, so no host control word (MXCSR / FPSR) is ever touched and the result is identical on every host and every build. SoftFloatBackend implements the full surface, IEEE and VAX alike, including the VAX F/G arithmetic, compares, and conversions that OpenVMS depends on.

 

An optional native fast-path backend (HostBackend) provides the seam for using host FP hardware where it is faster; it inherits SoftFloatBackend so any operation it has not specialized still delegates to the deterministic path, and its results are validated against SoftFloat (the fp_softfloat_vs_host_bench test exists for exactly this). The build-time EMULATR_FP_SOFTFLOAT option selects SoftFloat as the active backend; the longer-term aim is to expose the SoftFloat-versus-native choice as a runtime setting. Where a leaf still uses the host directly, the FPCR rounding and IEEE-trap behavior is realized with <cfenv> as described above; the SoftFloat path needs no such host-state juggling because it carries rounding and flags as data.
