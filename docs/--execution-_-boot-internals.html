Execution / Boot Internals

This appendix is the implementation-level companion to the Execution Flight Plan. It walks one instruction through V4's six pipeline stages, then describes how a fault is delivered to PALcode and how the firmware boot path is fed during the decompression loops. The code is in pipelineLib/PipelineDriver.h; the entry-vector table is in coreLib/Ev6EntryVectors.h.

 

The six-stage step

EmulatR runs a functional, single-slot pipeline: each instruction is carried by a PipelineSlot through all six stages in order before the next is fetched. The pipeline driver is the only component that mutates cpu.pc -- the leaves and the memory drainer never touch it.

 

Stage

Work

IF

Mint a slot from cpu.pc; translate the VA to a PA (via translateInstruction) and read the 4-byte instruction word. The fetch consults an optional IFetchOverride first.

DE

Dispatch lookup against grainFactory::g_primaryTable (plus the sparse secondary tables) to bind the decoded grain leaf.

GR

Build the ExecCtx: read the register file (integer or FP per the grain) into the execution context.

EX

Invoke grain.execFn(grain, ctx) and store the returned BoxResult onto the slot (any memory effect is packed here, not performed).

MEM

MemDrainer::drain(slot, cpu, memory) -- translate any data effect, perform the load/store, sign-extend loads, commit the register file, and resolve LL/SC.

WB

Advance cpu.pc from the grain's next PC, a divert target, or a fault entry vector; intercept HALT and faults; increment the cycle count.

 

The driver exposes step (one slot through all six stages) and run(maxCycles). The hot loop ships optimized; an EMULATR_DEBUG_STEP CMake option restores an -O0 build for single-stepping.

 

Fault delivery to PALcode

Faults travel as a faultCode on the slot's BoxResult and are resolved at WB by retire. HALT is a graceful stop, not a trap: it sets cpu.halted and returns. Any other non-zero fault is delivered to PALcode by the sequence in the EV6 HRM (6.8.1): save the faulting PC in EXC_ADDR (bit 0 carries the pre-trap PALmode so HW_REI restores it on return), map the fault to an entry-vector offset with coreLib::ev6::entryForFault, divert PC to palBase + offset, and enter PAL mode. The mapping covers the architectural classes -- for example UNALIGN at 0x280, OPCDEC at 0x400, DFAULT at 0x380, DTB single-miss at 0x300, ITB miss at 0x580, MCHK at 0x500 -- on the 0x80-spaced vector grid from 0x100 to 0x780. The absolute target is computeHwExceptionEntry(palBase, offset), which ORs the offset into the 32K-aligned palBase.

 

Delivery falls back to halt-with-diagnostic only when PALcode is not loaded (palBase == 0, where diverting would land in low-memory garbage and lose the failure) or when the fault has no architectural entry (kEntry_None). Routine high-volume TB misses (kFaultDtbMiss / kFaultItbMiss) are excluded from the fault log; everything else is recorded to logs/faults.log for offline review.

 

Feeding the boot path

During the firmware's self-decompression loops the IF stage consults an optional IFetchOverride before the normal translate-and-read, so fetches can be served from an immutable SRM payload while the guest rewrites memory underneath itself. The cold-boot inflate is the dominant early cost (see Decompressor Internals). Beyond decompression, the current runtime blocker is downstream: a clock-interrupt return through hw_ret R2 (the genuine instruction at PC 0xd954 confirmed by the host decompressor oracle), tracked on the Current Development Status page.
