Firmware Decode & Decompression

The shipped SRM image is compressed. At every cold boot the firmware decompresses itself on the emulated CPU, using the original DEC SRM decompressor -- a Huffman / inflate engine (Mark Adler's inflate, version c10p1) wrapped by a DEC decompress() routine. EmulatR runs that code faithfully rather than substituting its own.

 

What the startup does

The guest assembly startup (ev6_huf_decom.m64) is more than a call to decompress(): it first sets up CPU and PAL state -- establishes PAL_BASE, sets the I_CTL SDE bits, flushes the ITB / DTB and the instruction cache, primes the shadow registers, and saves / restores the SROM parameters passed in registers R16-R21 -- and only then inflates the image. Any future "skip the inflate" optimization must preserve that exact end-state.

 

Cost

The inflate inner loop is about 4 million cycles per cold boot (the spin observed at PC 0x60111c). That one-time cost is the main reason snapshots exist -- a saved post-decompression state lets you skip straight to the prompt on later runs.

 

The host oracle

To verify the decompressed bytes are correct without trusting EmulatR's own CPU (or a second emulator), the project builds the same DEC source as a native host program in tools/host_decompressor. Its output matches EmulatR's decompressed image byte for byte. This is a stronger reference than AXPBox's decompressed.rom, which runs the guest decompressor on AXPBox's own emulated CPU. The oracle settled the "single-bit PAL corruption" question: the hw_ret R2 instruction at PC 0xd954 is genuine firmware, not a decompression artifact.

 

A host-native decompression path (running the inflate off-CPU while preserving the PAL side-effects) is a possible future optimization; today the faithful on-CPU path is what runs.
