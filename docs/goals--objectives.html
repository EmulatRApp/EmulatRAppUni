Goals & Objectives

EmulatR is a full-system, cycle-based, high-fidelity emulator of the DEC Alpha AXP architecture. Its purpose is not to approximate an Alpha but to behave like one -- faithfully enough that the most demanding original software cannot tell the difference. This page sets out the single standard the project is held to and the priority-ordered goals every design decision serves.

 

The fidelity standard

EmulatR is governed by one overriding standard: it must behave like real Alpha hardware in every architecturally visible way, even when doing so complicates the implementation. This is applied uniformly across all subsystems -- the instruction set, memory translation, cache and coherency behavior, the PAL boundary, interrupt and exception delivery, device and DMA semantics, and multiprocessor ordering. A subsystem is correct only when no architecturally visible difference from hardware can be observed through it.

 

The test of fidelity is black-box and adversarial. The judges are not internal assertions but the most demanding guests available: the SRM console and PALcode exercise the privileged architecture directly, a booting operating system exercises the memory model, trap machinery, and device fabric under real workloads, and differential validation against a reference emulator checks behavior at the instruction and subsystem level. Fidelity here means architectural-semantic accuracy, not the reproduction of a particular chip's timing: EmulatR reproduces exactly what the architecture makes visible -- the cycle-based run loop, the precise point at which results and exceptions commit, the ordering guarantees and barriers, and the privileged state transitions.

 

Primary goals (in strict priority order)

EmulatR is designed around five primary goals. Every design decision is evaluated against them in this order; when two goals conflict, the higher-priority goal wins.

 

1.Architectural correctness first. The emulator models Alpha's architectural semantics, not its performance characteristics. Correctness is always prioritized over speed.

2.Deterministic and debuggable execution. Execution must be understandable, reproducible, and inspectable. One iteration of the run loop corresponds to one hardware clock cycle, enabling deterministic replay and debugging.

3.Explicit serialization. Ordering and synchronization are never implicit. All serialization is driven by architectural mechanisms -- the MB, WMB, EXCB, and TRAPB barriers, PAL entry, and exceptions -- and exceptions are delivered precisely at the architectural commit point, the Writeback stage.

4.SMP as a first-class concern. Multiprocessor behavior is fundamental to the design, not layered on afterward. Cross-CPU coordination is modeled with explicit barriers, reservation tracking, and IPI delivery.

5.Separation of concerns. Instruction decode, execution, memory access, coherency, privilege, and devices are isolated into well-defined functional domains (the "Boxes") with one-way dependency flow. No domain reaches into another's internal state.

 

Strategy: one set of objectives, four implementations

These objectives have been invariant across the entire history of the project. Four distinct implementations have been written; each changed the mechanism, never the goals. The current design (V4) is the first to satisfy all five at once -- it keeps a flat, shallow dispatch path for determinism and debuggability while resolving each instruction to a faithful, fully-architected execution leaf. The strategy throughout has been to let the implementation evolve toward the fixed standard, never to relax the standard to fit an implementation.
