Addressing

Virtual-to-physical translation is performed by Ev6Translator (mmuLib) exactly as the EV6 architecture specifies. In PAL mode the address bypasses translation (VA = PA). Otherwise the translator applies the canonical virtual-address window check and the superpage (kseg) mapping; an address that is neither a superpage nor a resident mapping returns an ITB or DTB miss for the PAL fill handler.

 

The translation buffers (SPAM)

The ITB and DTB are modeled by the SPAM cache -- Set-Partitioned Associative Memory -- as SPAMShardManager instances that live in CpuState (itbMgr and dtbMgr). Each is 16 shards by 8 ways = 128 entries: the manager hashes a lookup tag to pick a shard, and the shard does a short way-scan. The buffers are CPU-local by design -- each CpuState owns its own pair, and cross-CPU invalidation (in a future SMP build) would go through IPI machinery.

 

Invalidation is lazy (epoch-based)

Rather than walk and clear entries, invalidation bumps a counter (TlbEpoch): a full flush (TBIA) advances a global counter and a per-process flush (TBIAP) advances a process counter, so a flush is O(1) and a stale entry simply fails its liveness test at the next lookup. Targeted single-entry flushes (TBIS / TBISI / TBISD) clear one slot. The page-table-entry format (Ev6PteFormat) carries the protection, granularity-hint, and fault bits the guest expects, and translation faults are delivered precisely at the Writeback stage.
