Running EmulatR from the Shell

EmulatR is a console application driven from a command line. The standard execution shell is bash -- the project's own launch and trace scripts are bash, and the examples here assume it (on Windows, a bash shell such as Git Bash / MSYS, where the disk appears as /d/EmulatR). Everything below is run from the run directory.

 

The run directory

The run directory is the folder that holds the built executable. CMake's post-build step lays it out so the program and its inputs sit together:

 

<run>/ Emulatr.exe the emulator

 firmware/ SRM images (e.g. ds10_v7_3.exe)

 tools/ deployed bash helpers (trace_halt.sh, ...)

 EmulatrV4.ini settings (memorySize, paths, ...)

 ds10_flash.rom persisted NVRAM / flash

 snapshots/ saved machine states (.axpsnap)

 

Always launch from this directory (cd <run> first), because the default firmware path and the ini are resolved relative to the current directory. The bundled scripts in tools/ compute the run-dir root from their own location ($SCRIPT_DIR/..) and cd there before running.

 

Basic invocation

 

cd /d/EmulatR/<run>

./Emulatr.exe --firmware firmware/ds10_v7_3.exe

 

With no --firmware, the image falls back to [Rom] firmwareImage in the ini; with no --mem, memory falls back to [System] memorySize in the ini (1024 MB by default), and only then to the built-in 64 MiB default. The command line always wins over the ini.

 

Command-line switches

 

Switch

Meaning

--firmware <path>

Firmware image to load (resolved against the run directory).

--firmware-format auto|raw|srm

Override image-format detection (default auto -- SRM if the .exe signature scan succeeds, else raw).

--mem <bytes>

Guest memory size in bytes (e.g. 1073741824 for 1 GiB).

--load-pa <addr> / --start-pa <addr>

Physical load address and execution start address (hex or decimal; start-pa low bit sets PAL mode).

--max-cycles <N>

Cap the run loop at N cycles (default unbounded).

--pal-mode

Force PAL mode at reset.

--no-autoload

Suppress autoload of the newest snapshot -- forces a genuine cold boot.

--autosnapshot on|off

Toggle periodic auto_*.axpsnap saves (off avoids the disk cost on a long cold boot).

--snapshot-on-pc <pc[,pc...]>

Capture a one-shot snapshot when the CPU retires at the given PC(s).

--dump-disasm <pa>:<count>

Disassemble count instructions at a PA after load, before the run starts.

--trace <dec,machine>

Install the trace sink writing the decode and machine listing logs.

--log-disable / --log-only / --log-verbose / --log-file

Tune the log subsystem (see Logging below).

--help

Print the usage text and exit.

 

Logging

The simplest log is the console transcript, captured with tee so it is both shown and saved:

 

./Emulatr.exe --firmware firmware/ds10_v7_3.exe 2>&1 | tee run_$(date +%Y%m%d_%H%M%S).console.log

 

The log subsystem is filtered per channel. Show only certain subsystems, silence noisy ones, or send a channel to its own file:

 

./Emulatr.exe --firmware firmware/ds10_v7_3.exe --log-only ide,iic

./Emulatr.exe --firmware firmware/ds10_v7_3.exe --log-disable chipset

./Emulatr.exe --firmware firmware/ds10_v7_3.exe --log-file ide=logs/ide.log

 

Retire / instruction traces are written to the directory named by EMULATR_TRACE_DIR; EMULATR_TRACE_WINDOW=1 arms a windowed sink so a cold boot does not emit a multi-gigabyte stream. A typical traced cold boot:

 

export EMULATR_TRACE_DIR=/d/EmulatR/traces; mkdir -p "$EMULATR_TRACE_DIR"

export EMULATR_TRACE_WINDOW=1

./Emulatr.exe --no-autoload --firmware firmware/ds10_v7_3.exe 2>&1 | tee cold_$(date +%H%M%S).log

 

The full list of EMULATR_* runtime variables (console port, flash path, idle warp, trace controls) is on the Environment Variables page.

 

Running as a forked / background process

Because the console is a RAW TCP server (reached by a terminal client over EMULATR_CONSOLE_PORT), the emulator does not need the controlling terminal and can run detached. Launch it in the background, capture its output to a file, and note the process id:

 

./Emulatr.exe --firmware firmware/ds10_v7_3.exe > run.log 2>&1 &

echo "emulatr pid = $!"

 

To survive the shell closing, use nohup and disown, then connect from a separate terminal client:

 

nohup ./Emulatr.exe --firmware firmware/ds10_v7_3.exe > run.log 2>&1 & disown

# then, from anywhere: putty -raw 127.0.0.1 -P 10023 (or any RAW-TCP client)

 

Stop a detached instance cleanly by ending its console session or signalling the captured pid; avoid a hard kill during a flash write, which can corrupt ds10_flash.rom. To run two instances from one directory, give each its own flash and console port so they do not collide:

 

EMULATR_FLASH_ROM=flash_b.rom EMULATR_CONSOLE_PORT=10024 \

 ./Emulatr.exe --firmware firmware/ds10_v7_3.exe > run_b.log 2>&1 &
