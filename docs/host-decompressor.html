tools/Host Decompressor

tools/host_decompressor is a native, host-compiled build of the original DEC SRM Huffman / inflate decompressor. It serves as a trusted reference -- an oracle -- for the compressed Alpha firmware images, producing the decompressed firmware byte-for-byte without running any emulation. This page is the tool view (build, run, outputs); the algorithm and its place in the boot path are covered under Decompressor Internals.

 

Why it exists

The guest firmware self-decompresses on the emulated CPU at every cold boot (about 4 million cycles -- the spin at PC 0x60111c). To confirm that EmulatR's CPU produces the correct decompressed image without trusting either EmulatR's own CPU or AXPBox, the tool builds the same algorithm DEC used, from the same sources the firmware was compiled from, as a native host program. The reference image it produces matches EmulatR's emulated-CPU output byte for byte, which makes it both a regression guard and a clean way to mint a decompressed ROM. It is strictly stronger as an oracle than AXPBox's decompressed.rom, which is generated by running the guest decompressor on AXPBox's own emulated CPU and so still depends on a correct CPU and a contrived entry sequence.

 

Sources

Under src/, pinned copies of the read-only DEC originals plus the harness:

• inflate.c -- Mark Adler inflate, version c10p1 (the engine).

• decom.c -- the DEC wrapper (main + decompress); reference only, not compiled.

• decomp.h -- types and the ROM / non-ROM switches.

• ev6_huf_decom.m64 -- the guest assembly startup that calls decompress(); reference only (it also performs CPU / PAL state setup, which any in-emulator host path must preserve).

• oracle.c -- the harness: it locates the WimC header, sets the input / output pointers and compressed size, calls inflate(), and dumps the image.

 

Build and run

Build with the bundled script (build.sh on Linux / gcc, build.bat on Windows), then run the oracle against a compressed firmware image:

 

sh build.sh

./oracle <path-to-compressed-firmware.exe>

 

Output is written to ref_decompressed.bin with a printed summary; a known-good run of es45_v7_3.exe is cached under out/. The build_firmware_variants.sh helper drives the oracle across multiple firmware images (the DS10 image, for example, yields decompressed_ds10_v7_3.bin, which is the input the Ghidra workflow analyzes).

 

Compressed-image layout

The oracle finds the firmware payload by a linear scan for the WimC magic (0x436D6957). For es45_v7_3.exe the header sits at file offset 0x2400; its longwords give the compressed size (0x1a6164), the decompress target (0x8000), and the start of the compressed data (WimC + 20 = file 0x2414). The decompressed output is 0x30cc00 bytes (3,197,952).

 

The regression guard

In the decompressed image the V4-runtime quad at PA 0xd950 is 0x7be2a000_b4de0030 -- the hw_ret R2 at 0xd954 uses R2 (byte 0xe2), never R6 (0xe6). This is the signature that settled the 2026-05-30 single-bit-PAL-corruption question: that instruction is genuine firmware, not a decompression artifact. If a future change makes the guest CPU produce R6 in that neighborhood, it is a CPU regression, not a fix.

 

Integrating the oracle as a runtime alternate decompression path (--decompress=inline|host|cache) is deferred until the SRM reaches the >>> prompt; the recommended intercept-hybrid design is described under Decompressor Internals.
