Create Guest Disk Storage

EmulatR provisions disk images two ways: automatically at boot, driven by the platform manifest (create_if_missing), and manually with the tools/mkdisk.py generator. Both share one image-creation codepath and one geometry catalog, so a disk type's size is defined in exactly one place. A manifest references a disk type, not a raw byte count -- the size is resolved from the catalog at load.

 

Geometry catalog (single source of truth)

The authoritative geometry source is config/disk_types.json, generated at build time from dec_disk_media_types.tsv. Never hand-maintain both: edit the TSV, regenerate the JSON. Each type names total_lbn and block_bytes; capacity is derived (total_lbn * block_bytes). A type is either a real DEC model (for example RZ28) or a sized blank target (for example blank-4G).

Automatic provisioning (create_if_missing)

When a manifest storage entry names a missing image and sets create_if_missing, EmulatR provisions a blank sparse image of the catalogued size on first boot. It never overwrites an existing image, and never provisions read-only media (an ISO is supplied, not created). Example manifest entry for the dqa0 ATA disk:

 

{ "channel": 0, "unit": 0, "type": "ata_disk",

 "media": "dqa0.img", "disk_type": "blank-4G",

 "create_if_missing": true }

 

On first cold boot the run log reports the provisioning and the attach:

 

Storage: created blank disk image 'dqa0.img' (4294967296 bytes)

attached ATA disk 'dqa0.img' to IDE ch0 unit0

 

Manual creation (tools/mkdisk.py)

Use mkdisk.py to pre-make a typed image outside the boot path -- a specific DEC model, or a sized install target. It calls the same core as create_if_missing: looks up total_lbn * block_bytes from the catalog, writes a sparse zero-filled file, refuses to overwrite without --force, never creates for read-only media, and self-checks that the final size equals the catalog size.

 

python tools/mkdisk.py --type blank-4G --out dqa0.img

python tools/mkdisk.py --type RZ28 --out dka0.img

 

The image written is BARE: exactly total_lbn * block_bytes, zero-filled, with no partition table and no DEC disklabel. The guest OS install writes those structures. For the install path (boot the installer, install onto a blank target, boot the installed disk) a bare image is what you want.

 

Cross-platform sparseness

Image creation uses a truncate-to-size, which is genuinely sparse on Linux ext4 (a 4 GiB image does not write 4 GiB physically). On Windows NTFS the file is lazily zero-filled and is not flagged sparse unless FSCTL_SET_SPARSE is issued; mkdisk.py sets it so behavior is identical on Windows and Linux. In practice the image costs nothing on disk until the guest writes to it.
