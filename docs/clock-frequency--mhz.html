Clock Frequency and the MHz Reading

The SRM banner and show config report a CPU speed -- on the DS10 profile, 268 MHz. That number is not a hardcoded banner string and it is not read from a stored constant; the firmware derives it by measuring the CPU cycle counter against the chipset interval-timer interrupts. This page explains how that figure is produced and how to change it.

 

The profile clock

EmulatR has one build-time constant for the modeled CPU frequency, EMULATR_PROFILE_ALPHA_CLOCK_HZ, defined in chipsetLib/Tsunami21272_CsrSpec.h. The current DS10 value is:

EMULATR_PROFILE_ALPHA_CLOCK_HZ = 268435456 (2^28 Hz = 268.435456 MHz)

Everything downstream -- the interval-timer cadence and the frequency the firmware reports -- follows from this one number.

 

The interval timer cadence

The Cchip interval timer is designed to fire at the architectural nominal of 1024 Hz. The cycle target per tick is the profile clock divided by 1024, rounded to the nearest power of two so the fire test is a single mask compare:

kCchipIntervalTimerCycles = kProfileAlphaClockHz / 1024

kCchipTimerBit = roundLog2Nearest(kCchipIntervalTimerCycles)

fire when: (cycleCount & kCchipTimerMask) == 0 && cycleCount != 0

At 2^28 Hz this is exact: 2^28 / 1024 = 2^18, already a power of two, so the rounding introduces zero error and the timer fires every 2^18 cycles -- a clean 1024 Hz.

 

How the firmware turns that into MHz

The SRM measures its own clock: it reads the CPU cycle counter (PCC / RPCC) and counts how many cycles elapse across a known number of 1024 Hz interval-timer interrupts, then multiplies cycles-per-tick by the tick rate.

measured clock = cycles-per-tick x tick-rate

 = 2^18 cycles x 1024 Hz

 = 2^28 Hz = 268.435456 MHz -> "268 MHz"

So the reported MHz is the profile clock, recovered by measurement -- the calibration path comes back to exactly the value that drove the timer in the first place.

 

Why not the HWRPB constant?

The Hardware Restart Parameter Block does carry a cycle-counter frequency field (cycle_count_freq) and an interval-clock field (intrclock_freq, interrupts per second times 4096), built by HwrpbBuilder. But the cycle_count_freq spec value is 500 MHz, while the banner shows 268 -- so on the DS10 the firmware is clearly using its own measurement rather than the stored constant. (Firmware that instead trusted the HWRPB field would report 500.) The interval-clock field is set consistently with the 1024 Hz cadence.

 

Changing the reported speed

Override the profile clock at configure time, for example for an ES40 profile:

cmake -DEMULATR_PROFILE_ALPHA_CLOCK_HZ=600000000 (ES40, ~600 MHz)

One caveat follows directly from the calibration mechanism: a clock that is not a power-of-two multiple of 1024 makes kCchipIntervalTimerCycles round to the nearest power of two, so the tick rate drifts slightly from 1024 Hz and the firmware's measured MHz drifts with it (for ES40 600 MHz the tick lands at bit 19, about 1144 Hz, within ~12 percent of nominal -- harmless to a rate-agnostic OS but visible if a guest reads the calibrated speed). Powers of two like 2^28 calibrate exactly; other values trade a small reported-speed skew for the chosen cadence.
