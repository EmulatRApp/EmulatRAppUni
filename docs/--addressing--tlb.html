Addressing & TLB

This appendix is the implementation-level companion to the Addressing page. It walks the V4 translation path from a virtual address to a physical address, then describes the translation-buffer front end and the epoch scheme that makes a TLB flush a single counter bump. The authoritative reference is the Alpha Architecture Reference Manual (sections 4.1.6 / 4.1.7) and the 21264 Hardware Reference Manual; the V4 code lives in mmuLib/Ev6Translator.h and pteLib/.

 

The translator

Ev6Translator is not an object -- its entry points are static functions that take a CpuState const& as the first argument, so per-CPU state travels through CpuState rather than a translator instance. The MEM stage calls one of three entry points depending on the memory effect a leaf packed onto its BoxResult at EX:

 

•translateData -- a data load or store, no alignment check.

•translateDataAligned -- a data load or store with a size-class alignment check (1/2/4/8 bytes).

•translateInstruction -- an instruction fetch (longword-aligned, consults the I-side superpage enables).

 

Each returns a TranslationResult. On Success the caller's pa_out holds the physical address; any other value is a translation fault that the MEM drainer converts to a fault code (mmuLib::toFaultCode) and uses to short-circuit the register-file commit.

 

Translation path (data stream)

For a normal load or store, translateData proceeds in this order:

 

1.-Explicitly-physical accesses bypass upstream. HW_LD / HW_ST / LDQP / STQP are tagged physical and the MEM drainer sets pa = va before ever calling the translator. Importantly, EV6 PAL mode does NOT blanket-disable D-stream translation -- a normal load or store in PAL mode still translates. (An earlier blanket PAL bypass mis-mapped VMS system-space kernel-stack addresses and was removed.)

2.-Canonical VA check. VA_CTL bit 1 selects 48-bit VA mode when set, otherwise 43-bit mode. In 43-bit mode VA<63:43> must sign-extend VA<42>; in 48-bit mode VA<63:48> must sign-extend VA<47>. A non-canonical address faults before any TLB lookup.

3.-Kseg / superpage detection. The three SPE bits (on M_CTL for data, I_CTL for instructions) gate three kernel-only superpage modes. SPE[2] maps VA<43:13> directly to PA when VA<47:46> = 0b10; SPE[1] and SPE[0] cover the other two windows. A kseg-shaped address from non-kernel mode is an access violation; a non-kseg address falls through.

4.-DTB lookup. On a live hit, the PTE permission bits are checked and PA is composed as (PFN << 13) | (VA & 0x1FFF). On a miss the result is DtbMiss, which the drainer maps to kFaultDtbMiss; PALcode's DTB-miss handler refills the buffer (HW_MTPR DTB_TAG0 / DTB_PTE0) and the access retries.

 

The instruction path mirrors this with a 4-byte alignment check (PC bit 0 is the PALmode flag and is masked before the check), the I-side SPE bits, and an ITB lookup that misses to kFaultItbMiss. The EV6 base page is 8 KiB (shift 13); a PA is 44 bits wide.

 

The translation buffers (SPAM)

Each CpuState owns its own pair of SPAMShardManager front ends -- one ITB, one DTB. A manager holds an array of SPAMBucket shards, the epoch state, and a small hash that picks a shard from the tag. In CpuState the two managers are instantiated as SPAMShardManager<16, 8> -- 16 shards by 8 ways = 128 entries each, matching the architectural EV6 ITB and DTB size (the shard count and ways are template parameters, so the geometry can be retuned without touching the lookup path). Its public surface mirrors the architectural operations: lookup (one per access), insert (the miss handler / HW_MTPR ITB_PTE / DTB_PTE), the three invalidate operations, and clear for cold boot. Because translation buffers are CPU-local by construction, the manager carries no synchronization primitives; any future SMP cross-CPU invalidation would arrive through IPI machinery processed on the target CPU's own thread.

 

Epoch invalidation

A naive TLB would walk every entry on a flush. V4 instead keeps two counters in the manager and stamps both into every entry at insert time. A flush of all entries is then a single counter bump; entries become stale the instant their stamped value fails to match the live counter on the next lookup. The mapping from PALcode flush operations is:

 

Operation

Action

Lazy?

TBIA

bump globalEpoch (invalidates everything)

yes

TBIAP

bump processEpoch (ASM=1 global entries survive)

yes

TBIS / TBISI / TBISD

locate the matching entry, clear it in place

no (targeted)

 

The liveness test applied at lookup is: an entry is live when its globalEpoch matches the manager's globalEpoch AND either the entry is ASM-global or its processEpoch matches. The ASM arm encodes the architecture's rule that an ASM=1 (global) mapping survives TBIAP. The counters are plain 64-bit integers, not atomics -- at one bump per 100 ns a 64-bit counter takes roughly 58,000 years to wrap, so wrap is not handled; a manager reset zeroes both counters and clears every shard if it ever matters.
