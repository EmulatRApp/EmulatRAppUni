Guest Boot

From the SRM console prompt (>>>) you boot a device with boot <device> [-fl root,flags]. The device is a console device name; the optional boot flags select how the operating system bootstraps. Three SRM environment variables supply the defaults used when you type boot with no arguments.

 

Device names

dqa0 is the IDE primary master (the ATA fixed disk); dqa1 is the IDE primary slave (the ATAPI CD). dka0 is the first SCSI disk; ewa0 is the first Ethernet adapter (network boot). Booting the ATAPI CD (dqa1) is how an installer is started; booting dqa0 boots the installed system.

 

SRM environment variables (the boot register settings)

Set with set, inspect with show. These persist in NVRAM across reboots.

variable

purpose

example

bootdef_dev

default boot device for a bare boot

set bootdef_dev dqa0

boot_osflags

default OS boot flags (root,flags for VMS)

set boot_osflags "0,0"

auto_action

action at power-on: HALT, BOOT, or RESTART

set auto_action HALT

 

Boot flags -- OpenVMS

OpenVMS flags take the form root,flags. root is the system root (SYS0 .. SYSn, a single hex digit). flags is a hex bitmask; the load-bearing bit for interactive use is bit 0 (value 1), which requests a CONVERSATIONAL boot and stops at the SYSBOOT> prompt.

command

meaning

boot dqa0 -fl 0,0

normal boot from root SYS0

boot dqa0 -fl 0,1

conversational boot from SYS0 -- stops at SYSBOOT>

boot dqa0 -fl n,1

conversational boot from root SYSn

Other flag bits exist (debug, breakpoint, diagnostic-boot, memory-config display). The complete bitmask is platform-documented; consult the OpenVMS Alpha Upgrade and Installation manual. [TODO: confirm the full bitmask before documenting beyond bit 0.]

 

Conversational boot and SYSBOOT> (OpenVMS)

The bootstrap sequence: VMB.EXE (the primary bootstrap, on the boot disk) locates SYS$SYSTEM:SYSBOOT.EXE (the secondary bootstrap), loads it, and starts it. With the conversational flag set, the boot stops after loading SYSBOOT.EXE and displays SYSBOOT>. There you use SYSGEN SET and SHOW to adjust system parameters (the default parameter file on Alpha is SYS$SYSTEM:ALPHAVMSSYS.PAR), then CONTINUE to complete the boot.

 

>>> boot dqa0 -fl 0,1

 ... SYSBOOT>

SYSBOOT> SET WRITESYSPARAMS 0

SYSBOOT> CONTINUE

 

Boot flags -- Tru64 UNIX

Tru64 (OSF/1) also boots from the SRM console, but it has no SYSGEN conversational prompt; its analog is an interactive boot to the single-user shell, from which the kernel and startup can be controlled before going multi-user. Install from CD (the EmulatR acceptance path)

The standard install flow uses the ATAPI CD to boot an installer that writes onto the blank ATA disk, after which the installed disk is bootable:

 

>>> boot dqa1 (boot the installer from the CD / ISO)

 ... installer writes the system onto dqa0

>>> boot dqa0 (boot the installed system)

 

Requirement: a complete PCI interface

Reaching SYSBOOT> (and the OS device configuration that follows) requires the EmulatR PCI interface to be complete -- config cycles, BAR assignment, and dynamic BAR-to-range rebind. The SRM console probe is enough to enumerate dqa, but VMB, SYSBOOT, and the early OS adapter initialization traverse PCI config space to reach the boot device. Until PCI is complete, a conversational boot cannot be expected to reach SYSBOOT>. See Current Development Status.
