Decompressor Internals

This appendix is the source-level companion to the Firmware Decode & Decompression page. It identifies the DEC sources behind the cold-boot self-decompression, the CPU/PAL side-effects the guest startup performs, and the native oracle EmulatR uses to prove its emulated CPU produces the correct image. Treat the DEC SRM sources as authoritative; AXPBox is not.

 

What runs at cold boot

The guest firmware ships compressed and decompresses itself on the emulated CPU every cold boot -- roughly 4 million cycles, visible as the spin at PC 0x60111c. The image carries a WimC header (magic 0x436D6957) located by a linear scan; its longwords give the compressed size, the decompress target, and the start of the compressed data. For es45_v7_3.exe the header sits at file offset 0x2400, the target is 0x8000, and the decompressed output is 0x30cc00 bytes (3,197,952).

 

The DEC sources

The read-only originals live in Processor Support\Palcode\palcode\apisrm\apisrm\ref\:

 

•inflate.c -- the engine: Mark Adler inflate, version c10p1.

•decom.c -- the DEC wrapper providing main() and decompress().

•decomp.h -- types and the ROM / non-ROM switches.

•ev6_huf_decom.m64 -- the guest assembly startup that calls decompress().

 

The startup is more than a memcpy

The ev6_huf_decom.m64 startup is not a pure decompression: before and around the call it performs real CPU/PAL state setup -- it loads PAL_BASE, sets the I_CTL SDE bits, flushes the ITB / DTB and the I-cache, manages shadow registers, and saves and restores the SROM parameters in R16-R21. Any alternate "host decompress" path must preserve that end state, which is why a full bypass is risky and the recommended future design is an intercept-hybrid rather than a wholesale skip.

 

The native oracle

tools/host_decompressor/ builds the SAME algorithm DEC used, from the SAME sources the firmware was compiled from, as a native host program with no emulation involved. Its harness oracle.c replaces decom.c's main(): it finds the WimC header, sets the input/output pointers and compressed size, calls inflate(), and dumps the image to ref_decompressed.bin. The output matches EmulatR's emulated-CPU decompression byte for byte, which makes it a trusted regression guard. This is strictly stronger than AXPBox's decompressed.rom, which is produced by running the guest decompressor on AXPBox's own emulated CPU (and so still depends on a correct CPU and a contrived entry sequence).

 

The oracle settled the "single-bit PAL corruption" question: the instruction at PC 0xd954 (encoding 0x7be2a000 = hw_ret R2) is genuine firmware, not a decompression artifact. The known-good signature is the quad at PA 0xd950 -- a value using R2 (0xe2), never R6 (0xe6). If a future change makes the guest CPU produce R6 there, that is a CPU regression, not a fix.

 

Deferred: a host-native boot path

Integrating the oracle as a runtime alternate path (--decompress=inline|host|cache, default inline) is deferred until the SRM console reaches the >>> prompt. The recommended design is the intercept-hybrid: let the guest huf assembly run so its CPU/PAL side-effects happen for real, intercept entry to the decompress() C function, host-fill the output memory, set R0 to the decompressed base, and advance PC to the call's return site -- skipping only the multi-million-cycle inner inflate while inheriting every side-effect. A full bypass risks a subtle missing-side-effect divergence that would be painful to chase because the image bytes look perfect.
