grainFactoryLib

A grain is the unit of instruction semantics in V4: each decoded instruction resolves to a single grain leaf that implements its behavior. grainFactoryLib is where those leaves and the tables that reach them are generated, from tab-separated source-of-truth tables rather than hand-maintained switch statements.

 

Table-driven codegen

The codegen reads two tables and emits generated C++ headers and sources. SemanticFlags.tsv defines the semantic flag vocabulary; GrainMasterV4.tsv is the per-instruction master table. The generator codegen/genGrains.py validates the join between them and emits, under generated/: SemanticFlagsEnum.h (the flag enum, masks, and predicates), DispatchKinds.h (the dispatch-kind enum), GrainsForward.h (per-box leaf declarations), DispatchTables.cpp (the populated tables), and GrainStubs.cpp (stub leaves for not-yet-implemented rows). Hand-written leaves are carried in codegen/handwritten.tsv so a real implementation can supersede a generated stub.

 

The master table row

Each row of GrainMasterV4.tsv describes one instruction with these columns: mnemonic, primaryOpcode, subDecode, dispatchKind, format, box (the functional domain that owns the leaf -- IBox, EBox, FBox, MBox, CBox, or PalBox), flags (a bitwise-OR of S_* semantic flags), notes, and leafBase. The flags drive correctness-relevant behavior: for example a PAL row carries S_PalFormat | S_PalEntry | S_ChangesPC and one or more personality flags.

 

PALcode personalities

A single CALL_PAL opcode can mean different things under different operating systems, so rows carry personality flags -- S_PalTru64, S_PalVms, and S_PalLinux. For instance opcode 0x83 is CHMK (change mode to kernel) under VMS but callsys under Tru64 and Linux; the row marks all three. The codegen iterates the personalities to emit the right dispatch, which keeps the OS-specific PAL behavior in the table rather than in code.
