DispatchTables

The dispatch tables are how the Decode stage turns a raw instruction word into a grain leaf. They are generated into DispatchTables.cpp from the master table, so the decode path is data, not a hand-written switch.

 

Primary and secondary tables

The top-level lookup is g_primaryTable, indexed by the 6-bit primary opcode. Many Alpha opcodes are not fully decoded by that field alone -- arithmetic, floating point, the HW_MxPR instructions, and the PAL group all carry a sub-function in lower bits. For those, the primary entry points at a secondary (sparse) table keyed by the row's subDecode value, so the full decode is a primary index followed, where needed, by a sparse secondary lookup. The Decode stage performs exactly that: read the opcode, index the primary table, and follow the sub-decode where the kind requires it, yielding the grain leaf that the Execute stage will invoke.

 

Dispatch kinds

Each instruction is classified by a DispatchKind that names its decode family and selects which secondary table (if any) applies. The canonical set covers the integer families IntArith, IntLogical, IntMul, and IntShift; the floating-point families FltIeee, FltVax, FltLogical, ItFp, and FpTiExt; the privileged / hardware kinds HwLd, HwSt, HwMfpr, HwMtpr, HwRei, and Pal; and the control / miscellaneous kinds Direct, JmpClass, and Misc, plus a Reserved sentinel. A name helper (dispatchKindName()) is generated alongside the enum for trace and diagnostics.

 

Because the tables are generated, adding or correcting an instruction is a table edit plus a regeneration, not a hand change to decode logic -- the same source-of-truth discipline as the rest of grainFactoryLib. A row with no implemented leaf resolves to a stub (which logs and does not compute) or, if there is no row at all, the opcode takes a decode fault (OPCDEC).
