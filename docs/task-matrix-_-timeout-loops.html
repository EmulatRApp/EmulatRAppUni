DS10 -- Task Matrix & TimeOut Loops

Two pieces of DS10 engineering reference: the concrete bounds of the firmware's device-wait loops (useful when judging whether a long pause is a hang or a timed wait), and a snapshot of the open engineering work on the platform.

 

TimeOut loops

The firmware's IDE device-wait loops have concrete, bounded iteration counts. They are timed waits, not hangs; the longest is about half a minute of wall-clock per occurrence.

 

Loop function

Approx. wall clock

Notes

dq_wait_on_busy

5000 x 100 us

Wait for the IDE BSY bit to clear.

dq_wait_for_drq

5000 x 100--200 us

Wait for the data-request (DRQ) handshake.

dq_long_wait_on_busy

310000 x 100 us (about 31 s)

The long device-busy wait.

 

The EMULATR_IDLEWARP gate fast-forwards these idle waits without altering committed time. A separate, much longer pause -- the krn$_sleep-based option-firmware scan after "Checking dva0.0.0.0.0 for the option firmware files. . ." -- is a timed sleep rather than a device-wait loop and is a known performance item.

 

Task matrix (engineering status)

A snapshot of open DS10 engineering work, grouped by theme. This is a moving target; treat it as orientation, not a contract.

 

Networking. Promote the DE500 from a generic stub to a real DC21143 (tulip) model: enumerate ewa0 with a MAC read over the CSR9 SROM bit-bang, then add link / media select (SIA/MII) and the TX/RX rings for network boot.

Storage. Wire the media-backed read path (ATA READ 0x20 + ATAPI READ(10) to the file backend) so booting dqa0 works, and close the absent-unit IDE taskfile-read behavior.

PCI enumeration. Implement a real PCI bus walk so on-board devices are discovered and given valid BARs, replacing the firmware's all-ones fallback (the source of the repeated unhandled-outer-write probes).

Performance. Fast-forward the option-firmware scan sleep; keep the chipset off the hot path with a DRAM fast-path for fetch and load/store; event-driven console RX instead of a per-step poll; compile-guarded trace hooks.

Operational. Make system memory size resolvable from the QSettings ini; neutralize the "Halt Button is IN, BOOT NOT POSSIBLE" gate in software; centralize trace / diagnostic output under the build directory.
