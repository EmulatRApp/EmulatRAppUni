Memory Map & IPRs

This appendix is the register-level companion to the Memory and Addressing pages. It names the two internal-processor-register (IPR) namespaces EmulatR implements, the numeric indices a developer reads in a trace, and the physical-address regions the chipset arbiter routes. The authoritative reference is the Alpha 21264 Hardware Reference Manual (EC-RE2CA-TE, Table 5-4) together with V4's coreLib/HW_IPR.h.

 

Two IPR namespaces

EmulatR follows the architecture's split between registers the operating system reaches through CALL_PAL and registers only PALcode can touch with HW_MFPR / HW_MTPR.

 

1. PALcode-visible IPRs (0x0000 .. 0x00FF). These are CALL_PAL function codes, not hardware registers. The OS executes a CALL_PAL (for example MFPR_PTBR = 0x0015, MTPR_TBIA = 0x001B, MFPR_WHAMI = 0x003F); the PAL handler at the matching entry vector performs the actual access. Numbering follows the OSF/1 PALcode specification rather than the silicon. In V4 these are the PAL_MFPR / PAL_MTPR enums.

 

2. EV6 hardware IPRs (0x0100 .. 0x02FF). These are the real register indices encoded in the scbd field of an HW_MFPR / HW_MTPR instruction; only PALcode running in PAL mode may execute them. Numbering follows the 21264 HRM. The 0x0100 offset is an emulator convention so the hardware range cannot collide numerically with the PALcode-visible range: the decoder adds 0x0100 to the raw scbd value, except the PAL_TEMP block (raw 0x40..0x5F) which adds 0x01C0 so PT0 lands at 0x0200.

PAL temporaries

The EV6 provides 24 PAL temporary registers (PT0..PT23), reachable only via HW_MFPR / HW_MTPR in PAL mode. They are the backbone of PALcode: every PAL entry saves the guest GPRs into PAL_TEMPs and every PAL exit restores them. The OSF/1 convention pins specific temporaries to architectural state exposed through CALL_PAL (PT0 = KSP, PT2 = PTBR, PT4 = PCBB, PT5 = PRBR, PT6 = SCBB, and so on). The OS never reads a PAL_TEMP directly -- it issues CALL_PAL MFPR / MTPR and the handler relays to the right temporary. V4 widens the range to PT31 because the SRM firmware used during bring-up carries EV5-vintage code that addresses PT24..PT31 even on EV6 (observed at PC 0x12f88 issuing hw_mtpr Rx, 0x5F); those extra eight are pure CpuState-backed scratch.

 

IPRs that map to architectural state live as flat fields on CpuState (ptbr, i_ctl, pal_base, ps, excAddr, fpcr). Selectors that are not yet CpuState-backed return kFaultUnimplemented; adding one later is a CpuState field plus a switch case in the HW_MFPR / HW_MTPR leaf.

 

Physical address regions

Guest physical addresses are routed by the Tsunami chipset arbiter. Reads and writes resolve to one of three destinations: main memory (the GuestMemory sparse backing, bounded by the Cchip AAR registers -- anything outside is a non-existent-memory / NXM fault), the chipset CSR space (Cchip / Dchip / Pchip control registers and the TIG region), or PCI / device MMIO windows. The halt-gate diagnostics read and write CSRs in this space; for example the console arming read targets pmem:80130000FF8 in the TIG region. The exhaustive per-register CSR table is in the Tsunami / 21272 Hardware Reference Manual; the operator-facing view of how to read these locations is on the Evaluate Address page.
