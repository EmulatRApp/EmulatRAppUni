Firmware Hangs, Slow Boot, and Missing Devices

Firmware hangs (usually timed loops, not crashes)

Some long pauses are time-based firmware loops rather than emulator hangs:

•-The option-firmware scan -- after "Checking dva0.0.0.0.0 for the option firmware files. . ." the firmware enters a krn$_sleep-based wait (the 0x7bef0 software-tick loop). It is a timed sleep, not a device probe; a profiled gap of roughly 36 minutes has been observed here (open item).

•dqa0 "no media present" is followed by a stall (suspected timer loop).

•GCT/FRU initialization can be significantly delayed.

 

Slow boot (shorten it)

Boot time is dominated by power-up diagnostics and several bounded timeout loops. To speed iteration, turn the heavy diagnostics off at the prompt:

 

>>> set memory_test none

>>> set full_powerup_diags off

 

Bounded device-wait loops (approximate wall-clock): dq_wait_on_busy ~5000 x 100us; dq_wait_for_drq ~5000 x 100-200us; dq_long_wait_on_busy ~310000 x 100us (about 31 s). The EMULATR_IDLEWARP gate fast-forwards idle waits without altering committed time.

 

Missing devices

If a device is absent from show device, check the IDE / ATAPI enumeration path. The dqa CD enumeration gap is a known open item: the firmware does not always observe the device during the option-firmware scan. Tracing 0x1F0..0x1F7 plus the PCI config of bus 0 / dev 5 / func 1 distinguishes a port-never-initialized case from a probe-protocol mismatch.
