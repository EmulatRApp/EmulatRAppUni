# Session Journal -- 2026-05-08

## Summary

Built a per-retire diagnostic stream into V4's `DecListingSink` to
chase an apparent overnight wedge during SRM boot.  The new stream
emits one elided line per retired instruction (cycle, PC, mnemonic,
PALmode, exc_addr, plus only the integer registers whose current
value is non-zero) into a third file at `X:\traces\<stamp>_srm.trc`.
Mask bit `TRACE_RETIRE_COMPACT = 0x80` armed automatically by the
`--trace` CLI flag.  Folded in the long-pending debugger-pokeable
trace-window counter while we were touching the sink.  Wrote a
Python analysis script (`tools/analyze_retire_trace.py`) that
finds dominant PC loops in the trace.  Started a long overnight
run with a 200 GB kill-switch to capture the wedge state for
mining in the next session.

Net result: the V4 pipeline now produces machine-readable retire
traces suitable for hang/loop diagnosis without rebuilding, and
the harness for analyzing them is in place.  No actual wedge
diagnosis yet -- observed runs were in legitimate forward progress,
not stuck in a loop.

## Context

V3's overnight runs were "presumably looping" with no signal for
where.  V4 was initially missing a per-retire trace channel
suitable for grep / programmatic analysis, so we added one.  The
existing dec/machine channels in `DecListingSink` are designed
for full-instruction listings; they over-emit (32 register fields
per line whether non-zero or not) and don't decorate with
PALmode / exc_addr.  Wanted a third channel optimized for "find
the loop body."

## Decisions

- **Output path:** default `X:\traces\YYYYMMDD-HHMMSS_srm.trc`
  (Tim's 2 TB iSCSI reserve), override via
  `EMULATR_RETIRE_TRACE_DIR` environment variable.  iSCSI proved
  unreliable in the MSVC debug session so the envvar is the
  practical workaround.
- **Line format:**
  `RET cyc=<n> pc=<hex16> <mnem> pal=<0|1> exc=<hex16>[ R<dd>=<hex16>]*`.
  Mnemonic added per Tim's request after the first run made the
  bare PC hard to read.
- **Elision rule:** integer registers whose value is 0 are
  omitted; R31 omitted always (architecturally hardwired zero);
  F-registers omitted entirely (boot/SRM is FP-free).
  "Missing on this line" therefore means "zero on this cycle."
- **No IPL field:** V4's `CpuState` does not yet carry an
  architectural IPL.  Added a header comment noting the format
  will gain `ipl=<hex2>` once `CpuState` does.
- **Sync ofstream, no async logger:** every line ends with
  `flush()` so a crash mid-run does not lose recent context.
  Throughput penalty is acceptable -- this is a diagnostic
  stream, not steady-state production.
- **Default-on with `--trace`:** parser arms `0x40 | 0x80`
  (PAL_WINDOW + RETIRE_COMPACT) when `--trace` is supplied.
  Avoids requiring a new CLI flag for the v1 of this work.
- **Debugger-pokeable peek counter:** static
  `DecListingSink::s_traceWindowCountdown` (atomic int64).
  Operator deposits N from VS Immediate window to emit the next
  N retire lines regardless of mask, decrements per emitted line,
  self-disables at zero.  Closes the long-pending TODO in
  memory.

## Findings from runs

The retire trace surfaces real structural information about
SRM boot:

- **Cycles 0-30:** PALcode self-relocation at 0x900000 (BIS,
  BSR R02 -> 0x900060, LDAH+LDA constant synthesis, SUBQ to
  compute kseg-mapped addresses, including the canonical
  `0xffffffff_e0900000` form).
- **Cycle ~5M:** Already in non-PAL firmware code at PC
  ~0x600934.  Code making forward progress.
- **Cycle 6,101,439:** First chipset-region access -- a Pchip0
  sparse-I/O write at PA `0x801643F10F0`.  Decodes to a single-
  byte PCI I/O write.  This is SRM probing PCI devices via the
  sparse window.
- **Cycle 11,880,233:** First Cchip CSR access, write to
  `0x801A2070000` (offset 0x02070000 within Cchip CSR space --
  unusually high; documented Cchip registers all sit in the
  first 0xCC0 bytes of the window).  Worth investigating
  whether this is a CPU-mirror / per-CPU-bank decode that
  V4's `TsunamiCchip` does not yet implement.
- **Cycle 22M+:** Inside a tight 11-instruction byte-copy loop
  at PCs 0x600920..0x600948 (LDQ_U / LDQ_U / SUBL / EXTBL /
  INSBL / LDA / MSKBL / LDA / BIS / STQ_U / BGT).  Loop counter
  R18 decrementing; not a hang, just slow byte-by-byte copy.
- **Cycle 28.7M-28.92M (analysis window):**  Hottest 11 PCs
  account for 22% of records (the byte-copy loop, ~4,000
  iterations).  Outer driver loop ~64 instructions long at PCs
  0x6010xx, calling a worker subroutine at 0x600990 via BSR.
  Iterating ~1,500 times based on hot-PC counts.  No wedge
  signature -- a real wedge would show one PC at 80%+ or a
  small period repeating tens of thousands of times
  consecutively with frozen registers.
- **Pchip CSR writes (`0x80180000000-0x80190000000`):** zero
  in the captured window.  SRM has not yet configured the PCI
  bridge by cycle 28.92M.  Either the run was killed before
  reaching that phase, or there is an earlier hang we have not
  yet captured.
- **48 sparse-I/O writes total** in the captured window.  PCI
  device probing is happening but is not the dominant work.

## Evening session -- Tsunami chipset routing design

While the overnight trace was running, Tim drafted two proposal
documents in `Journals/`:

- `Session_Proposal_for_Consideration_Tsunami.txt` -- Tsunami /
  Typhoon chipset interface architecture.  Argues for a
  `PhysicalBus -> TsunamiChipset -> {CChip, DChip, PChip0,
  PChip1}` ownership tree with `GuestMemory` staying RAM-only,
  hose ownership inside the chipset, and a hot-path RAM bypass
  so chipset decode does not sit on every memory access.
  Includes Addendum A (hot-path performance considerations)
  recommending direct member access over virtual dispatch and
  `AXP_HOT` / `AXP_ALWAYS_INLINE` on the routing methods.
- `MachineChecks.txt` -- machine-check policy.  Argues for
  adding `kFaultMachineCheck` to the V4 fault enum *now* with
  Phase-1 soft handling (log, return 0, continue), Phase-2
  metadata tracking, Phase-3 full Alpha trap delivery
  (EXC_ADDR / MM_STAT / MCHK vector / pipeline flush / PAL
  entry).  Distinguishes recoverable (unimplemented MMIO
  inside a known device) from fatal (unmapped PA off the
  bus entirely).  Documents Cchip 0x02070000 as a likely
  per-CPU mirror stride rather than bogus address formation.

We discussed both and converged on a coherent policy.  The
agreement closes a real gap surfaced by today's trace mining:
SRM is writing chipset PAs (`0x801A2070000`, `0x801643F10F0`)
that have nowhere to land in V4 today -- either silently
dropped, returning OutOfRange, or wrapping into RAM.  Whichever
of those it is, SRM is talking to a void and proceeding on
whatever value it reads back.  The current byte-copy phase
visible in the trace may be either legitimate work or a
degraded path SRM falls into when chipset readbacks are
synthetic.

### Architectural decisions (locked for Phase 1 implementation)

1.  **PhysicalBus regions** = RAM + CHIPSET + fallthrough(FATAL).
    Three regions only; no separate ROM/INVALID branches.
    INVALID is the fallthrough -- not a peer.
2.  **ROM** = deferred for Phase 1.  When added, it lives inside
    `Pchip0` as the flash window at `0x1FC000000000`, not as a
    top-level `PhysicalBus` region.  V4's `SrmLoader` already
    copies the SRM image into RAM at PA `0x900000` plus a `0x0`
    mirror, so SRM never executes from the flash alias during
    boot today.
3.  **Fallthrough policy** = log + halt + `kFaultMachineCheck`
    FATAL.  Out-of-range PAs halt the run with full context in
    the post-mortem dump (PA, cycle, mnemonic, fault code).
    This is the diagnostic win for the overnight wedge: if SRM
    is silently spinning on out-of-range readbacks today, the
    next run halts the moment the first one happens.
4.  **`kFaultMachineCheck`** = added to V4 fault enum now.
    Phase-3 trap delivery is deferred but the type exists so
    fault handling does not get spread across hot-path logic
    (`OutOfRange` / `UnhandledMMIO` / `TODO` / `UnknownPA`).
5.  **Bus returns response, MEM stage delivers fault.**
    `MemoryResponse { success, data, fault }` shape.  No
    `throw`, no `cpu.enterPAL()` from the bus.  Pipeline owns
    the state transition.  Phase 1 = halt; Phase 3 = divert to
    PAL machine-check vector with EXC_ADDR / MM_STAT.
6.  **Three-layer fault policy** with deliberate asymmetry:
    - PhysicalBus: unknown region -> FATAL halt
    - TsunamiChipset: unknown subregion within chipset MMIO
      window -> FATAL halt
    - Per-chip handler: known window, unknown register
      offset -> RECOVERABLE (log warning, return 0, continue)
    The bus and chipset *fabric* halt because they should know
    what's wired to them.  The chips tolerate
    unknown-register-in-known-window because real SRM probes
    optional registers it does not strictly need.
7.  **`FaultInfo`** struct populated at the fault site with
    {code, pa, access, size, write, speculative}.  Captured now
    even though `speculative` will not be populated until cache
    prefetch lands -- preserves precise-fault metadata for
    Phase-3 EXC_ADDR / MM_STAT generation.
8.  **Tsunami internal regions for Phase 1** = Cchip + Dchip +
    Pchip0 (CSRs and sparse I/O decode) + ISA bridge stub +
    UART.  Pchip1 deferred to ES45 work.  Sparse decode lives
    inside Pchip using V4's existing
    `Tsunami21272_RegisterMap.h` helpers.
9.  **`TsunamiCchip` Phase 1 strategy** = strict documented CSR
    register set, unknown offsets log + return 0 (do NOT
    silently mask the address down to a known register).  This
    surfaces the 0x02070000 access pattern in instrumentation
    so we can decide whether it is a real per-CPU mirror or a
    one-off probe.  Per Tim's `MachineChecks.txt` Response 2.
10. **`IChipset` abstraction** = deferred until Typhoon work
    materializes.  Concrete `TsunamiChipset` for now; refactor
    to interface only when a second chipset is real.
11. **Routing trace channel** = new mask bit
    `TRACE_BUS_ROUTE = 0x100`, fourth ofstream in
    `DecListingSink` writing `<stamp>_bus.trc` under the same
    `X:\traces` directory (or the
    `EMULATR_RETIRE_TRACE_DIR` override).  Output format:
    `BUS R/W cyc=<n> pa=<hex16> target=<RAM|CCHIP|DCHIP|PCHIP0|PCHIP1|FATAL> [size=<n>] [csr=<name>]`.
    Cross-references with the retire-compact stream by cycle
    so a divergent route lines up with the instruction that
    issued it.
12. **Hot path** = `if PA in RAM range -> GuestMemory` is the
    first compare in `PhysicalBus::route()`, predicted taken,
    no virtual dispatch.  Other regions use direct member
    access into the composed Tsunami / per-chip objects.  No
    `IMemoryMappedDevice*` indirection on the hot path.

### Phased implementation order (also from the proposal)

**Phase 1 -- Reach SRM `>>>` prompt.**

`PhysicalBus` (RAM + CHIPSET + FATAL fallthrough); `TsunamiChipset`
(Cchip + Dchip + Pchip0 with CSRs and sparse I/O decode + ISA
bridge stub + UART path).  `kFaultMachineCheck` enum +
`MemoryResponse` shape + halt-and-dump on FATAL.  Routing
trace channel.  Expected outcome: SRM relocates, accesses
chipset, emits console output, reaches `>>>`.

**Phase 2 -- PCI enumeration.**

PCI config space, sparse memory windows, dense memory windows.
Fake enumeration responses initially.  Expected outcome: SRM
device discovery (SCSI, NIC, VGA visible to firmware).

**Phase 3 -- OS boot.**

Interrupts, DMA, timer behavior, hose ownership, translation
windows, full machine-check trap delivery (EXC_ADDR / MM_STAT
/ MCHK frame / PAL entry), per-CPU mirror decode in `Cchip` if
SRM access patterns proved it real in Phase 2 instrumentation.
Goal: OpenVMS / Tru64 boot.

### Why this matters for the overnight wedge investigation

Today's trace shows SRM writing to `0x801A2070000` (cycle
11.88M) and `0x801643F10F0` (cycle 6.1M) without any router in
place to handle them.  Whatever V4 is doing with those writes
today -- silent drop, `MemStatus::OutOfRange`, or RAM wrap --
SRM is reading values back from a fabric that does not exist.
The byte-copy loop visible at PCs `0x600920..0x600948` may be
legitimate firmware setup, or it may be a degraded code path
SRM enters because chipset readbacks are synthetic.  We will
only know after the routing layer is in place.

The Phase-1 FATAL halt on truly unmapped PAs is the proximate
diagnostic: the next run with this work in place will halt at
the first out-of-range access with the offending PA + cycle in
the post-mortem.  That is probably the wedge cause, exposed in
one rebuild.
