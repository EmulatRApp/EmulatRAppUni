## Context

V3 reached SRM-to-PAL handoff but hit a chain of bugs that resisted
incremental fixes:

- BoxResult `merge()` / `operator|` propagating flags without payloads
  (BOX_PC_COMMITTED set, architecturalPC dropped); fixed in V3 but the
  fix exposed nearby latent issues.
- SrmRomLoader `done:` predicate exiting before pipeline drain;
  required two `singleStep()` calls to fix without breaking the
  PAL relocation timing.
- BIS at 0x6005c0 not committing R30 to the architectural register
  file -- diagnosed today as the BIS grain dispatching to
  `executeHW_MTPR` (likely vptr suspicion or a generator template
  defect; the per-grain-class architecture made the failure mode
  silent).

The accumulated coupling between `BoxResult` flag propagation,
pipeline ring rotation, grain dispatch, and pipeline drain semantics
made each fix surface adjacent bugs of the same family.  Tim called
the pivot.

## Direction

V4 framing: best-effort deterministic architecture design.  Single
OS thread, four simulated AlphaCPU cores (ES45 target), CPU 0 only
exercised in v1.  Spec-driven: write the contract first, build to
it.  Modern C++ (C++20, MSVC) with Qt sparingly for utilities (Core
by default; threading / networking / IO when clearly the best tool;
core emulator is Qt-free).

Lessons from V3 encoded as structural constraints rather than
implicit conventions:

- POD `InstructionGrain` with function-pointer dispatch -- no
  vtable.  The "BIS dispatching to executeHW_MTPR" failure mode
  cannot be expressed.
- Function table generated from `GrainMaster.tsv` (V2 pattern) --
  no per-grain class headers, no auto-registration singletons.
- `BoxResult` pattern preserved (V1 pattern) for clarity at call
  sites, with three structural guards on the V3 merge defect:
  paired setters, tested propagation registry, paired
  documentation.

## Architectural decisions (all settled today)

### Execution model (`SPEC_execution_model.md`)

Six-stage pipeline (IF / DE / GR / EX / MEM / WB).  Architectural
register writes happen ONLY at MEM via `commitPending`.  PC redirect
ONLY at EX.  Slot in MEM is never squashed.

`InstructionGrain` is a POD with `void (*execute_fn)(PipelineSlot&)`
function pointer.  Target size <= 64 bytes (one cache line).

Two-level dispatch: `g_opcodeTable[64]` of `OpcodeEntry`, each
holding a `DispatchKind` and a sub-table sized by the format's
function-code width.  Operate format = 128 entries, PAL = 256, Jump
= 4, etc.  Storage in kilobytes, not megabytes.

PAL personality disambiguation via `_64` suffix on mnemonics (Tim's
naming convention from `GrainMaster.tsv`).  Codegen partitions rows
by suffix; OpenVMS rows lack the suffix, Tru64 rows carry it.

`BoxResult` shape with paired setters
(`setCommittedArchitecturalPC`, `setIntCommit`, `setBranch`, ...),
a propagation registry that drives `merge()` and is exercised by a
unit test, and per-flag documentation.  V3's merge defect cannot
recur.

R31 / F31 zero-write skip happens in `commitPending` at MEM, in
one place.  Executors stage commits unconditionally; commit applies
the architectural rule.

Separate integer and FP bypass tables (`g_intBypass`, `g_fpBypass`)
since EV6 has independent forwarding for the two register classes.

Store operands latched at GR; no re-read at MEM.

Boot-time dispatch fast-path optimization deferred until profiling
proves a hotspot.

Test harness scope: (Grain + Box + BoxResult) triplet, exercising
the wiring as well as the body.

Snapshot save/restore designed in spec (section 11) but
implementation deferred post-v1.

### Memory realms and post-JSR scaffold (`SPEC_memory_realms_and_scaffold.md`)

Realm map captured (DRAM low, PAL base region at 0x600000, boot
ROM staging at 0x900000, etc.) with `(inferred)` annotations and
`tbd-need-hrm-cite` flags where the source is V3 traces rather than
the ES45 HRM directly.

Post-JSR initialization touchpoints documented: PALcode entry stub
(v1-full), HWRPB (v1-stub), Tsunami chipset (v1-stub for CSRs,
deferred for full), IPRs (v1-full storage, v1-stub for cache-flush
side effects).

Scaffold dependencies: fault dispatch path (v1-full), CALL_PAL
routing (v1-full infrastructure, v1-stub PALcode handlers),
HW_MTPR side effects (PAL_BASE / TLB / ASN / IPIR / MCES),
HW_REI semantics, pipeline-level scaffold for trap-vs-retire
ordering.

**SPAM carryover decision: Option A (carry forward V3 SPAM
wholesale).**  Project target is 4-CPU ES45; multi-CPU machinery
is load-bearing for the project even though v1 only exercises
CPU 0.  Pruning multi-CPU support for v1 and reintroducing for v2
risks subtle correctness drift; carrying forward keeps the
multi-CPU semantics in place from day one.  Files included:
`alpha_spam_manager.h`, `alpha_spam_bucket.h`, `alpha_spam_types.h`,
`SPAMEpoch_inl.h`, `alpha_pte_core.h`, `AlphaPTE_Core.h`,
`alpha_pte_traits.h`, `Ev6PTETraits.h`, `Ev6SiliconTLB.h`,
`global_Ev6TLB_Singleton.h`, `buildCanonicalTLBTag_inl.h`,
`calculateEffectiveAddress.h`, `ev6Translation_struct.h`,
`TLB_Helpers_inl.h`, `TLBShootdownTypes.h`.

Multi-CPU bring-up itself (CPUs 1-3 active, OS handoff, full
Tsunami emulation, OS-specific features) is deferred past v1.
