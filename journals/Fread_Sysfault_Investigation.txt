2. WHY A FULL COLD-BOOT TRACE IS THE WRONG TOOL

  The SYSFAULT is at ~cyc 21.47B.  A full retire trace from cyc 0 would
  be terabyte-class (prior break.trc files hit 194 GB and 441 GB).
  Tim's proposal (2026-06-03) is correct: replay from a near-frontier
  snapshot and gate the trace to the build_dsrdb window -- ~10-30M
  instructions at most, Ghidra-ingestible.

3. WINDOWED-TRACE RECIPE

  Step A -- anchor snapshot (replay, do NOT re-grind 21B cycles):
    The periodic autosaver writes snapshots\auto_*_<cycle>.axpsnap every
    ~10M cycles.  Pick the newest one strictly BEFORE the SYSFAULT, e.g.
    auto_*_21400000000.axpsnap (cyc 21.40B).  Autoload picks the latest
    by mtime, so either (a) point the run at a snapshots dir containing
    only that file, or (b) copy it to a predig_ prefix (non-pruneable,
    autoloaded first).  Replays to the frontier in seconds.

  Step B -- gate full-complement trace to the build_dsrdb window:
    BreakpointSink gate:
        open  = 0x7d048   (build_dsrdb entry)
        close = 0x1c62d8  (sysfault entry -- fault caught)
        revolutions = 1   (single pass; build_dsrdb runs once)
    Override via s_gateOpenPc / s_gateClosePc / s_revolutionsRemaining
    before the run (the BreakpointSink already supports this; see
    reference_breakpoint_sink).  This captures every retired instruction
    from build_dsrdb entry through the fread fault and stops at the
    handler -- full GPRs + ea/val per memory op.

  Step C -- enable the standard --trace as a coarse backstop only if the
    gated capture misses the entry (e.g. build_dsrdb already running at
    snapshot time -- pick an earlier snapshot if so).

  Determinism guard: the SYSFAULT cycle is identical across the seeded
  and unseeded runs, so the replay is reproducible; if the windowed run's
  fault cycle differs from 21.47B, the snapshot/gate is off, not the bug.

4. WHAT TO EXTRACT (the questions the window answers)

  Q1. The exact faulting instruction: PC, encoded, op, and the fault code
      V4 raised (ACV / alignment / OPCDEC / DtbMiss-that-didn't-fill).
      -> the last BRK line before the gate's close marker.
  Q2. The faulting effective address (ea= field) and whether it is a VA
      that should map (DRAM) or an over-range / null pointer.
  Q3. What fread was reading: walk back from the fread entry (0x5a510)
      to the FILE* argument build_dsrdb passed (R16 at the call site
      0x7d0dc).  fread dereferences fp->ip->... ; a null or garbage fp
      member produces a bad-VA load -> ACV.  This mirrors the earlier
      fclose(&spl_kernel) class of bug: a FILE built from an unpopulated
      device/inode.
  Q4. WHICH file/device build_dsrdb opened: trace the fopen (0x59cd8)
      that produced the fp, and the name string it was handed.  Source
      pc264.c build_dsrdb has a DS20E branch that does
      fopen("iic_rcm_nvram0","srb") + fseek(0x11) + fread -- the shipped
      DS10 image may take an analogous nvram/eerom path.  If the file is
      an unbacked NVRAM/eerom device, the fp's data pointer is null and
      fread walks off it.

5. GHIDRA HANDOFF

  With the window .trc + the symbol table, in Ghidra:
    - Rebase to the decompressed image (load base 0x8000; file = PA-0x8000).
    - Apply the embedded symbol names (ApplySrmSymbols workflow,
      reference_srm_ghidra_symbol_recovery): name[i] <-> desc[i-1],
      entry = *(desc+8).
    - Disassemble build_dsrdb (0x7d048): identify the fopen target string,
      the fseek offset, and the fread destination + count.  Cross-check
      against pc264.c build_dsrdb (version drift expected; image wins).
    - The faulting load's source register chain (from the .trc) maps to a
      fp->member deref; Ghidra names the struct field.

6. LIKELY OUTCOMES + PRE-AGREED DIRECTION (decide AFTER the trace)

  H1  fread faults because build_dsrdb opened an UNBACKED NVRAM/eerom
      device (fp data ptr null) -> the fix is to back that device (like
      the eerom/flash work), NOT to model TIGbus reset.  Most likely
      given the fopen-driven path and the fclose(&spl_kernel) precedent.
  H2  fread faults on an over-range VA computed from a bad header byte
      read out of flash/nvram -> data-content fix (correct nvram seed).
  H3  A genuine V4 exception-delivery bug (ACV/alignment mis-raised on a
      legal access) -> pipeline/PAL fix; least likely (boot got this far
      cleanly), but the window's fault-code field settles it immediately.

  Only AFTER the window names the cause do we choose; the TIGbus reset
  model (machinery-map gap G8) is deferred -- it is the firmware's
  reaction to the fault, not the cause, and modeling it would only let
  the box "reset" in a loop rather than reach >>>.
